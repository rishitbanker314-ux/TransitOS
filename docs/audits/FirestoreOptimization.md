# Firestore Optimization Audit

**Target**: TransitOps Vehicle Management Module
**Scale**: 100,000+ Vehicles, 500 Concurrent Users

## Executive Summary
At 100,000+ records, Firestore cost and hot-document limits become the primary architectural constraints. The current implementation uses advanced patterns to mitigate these.

## 1. Aggregation & Dashboards (Pass)
**Strengths:**
- Dashboard relies on `analytics/fleet_snapshot`. This turns 100,000 read operations into 1 read operation per dashboard load.
- Estimated Savings: $150+/month at peak concurrency.

**Action:** Ensure the Cloud Function populating `fleet_snapshot` runs on a chron job (e.g., every 5 minutes) rather than triggering on every vehicle write to avoid a hot document write bottleneck.

## 2. Transactions & Concurrency (Pass)
**Strengths:**
- `VehicleStatusEngine.ts` utilizes atomic transitions.
- This guarantees status consistency across the fleet.

**Action:** Max limit for transactions on a single document is ~1 write per second. Vehicles rarely exceed this, but keep in mind for bulk updates.

## 3. Indexing Strategy (Needs Action)
**Weaknesses:**
- Complex compound queries (e.g., `status == "Active" AND manufacturer == "Volvo" ORDER BY mileage DESC`) require composite indexes.
- **Action:** Before deployment, capture all query shapes generated by the UI and AI filters, and explicitly deploy a `firestore.indexes.json` file.

## 4. Document Sizes (Pass)
**Strengths:**
- Vehicle documents are lean. Large historical data (trips, maintenance logs, audits) are segregated into their own collections or subcollections.
- Avoids hitting the 1MB document limit and reduces payload size on reads.

## Conclusion
The data model is highly optimized for read-heavy operations. Ensuring background aggregations and composite indexes are deployed will guarantee infinite scale.
